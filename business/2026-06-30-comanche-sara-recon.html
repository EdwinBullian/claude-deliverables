Part 1 — On-Site Recon Playbook Do in 48h

A · The data-exit question (highest priority)

This decides whether any AI/automation layer is even buildable. Comanche is closed third-party software — you will not modify it. So the layer reads its outputs. Find out what outputs exist:

  • What database does it run on? Go to the server/back-office PC. Check the Comanche install folder (e.g. C:\Comanche) and note file types — older Thai PMS installs often run on Sybase SQL Anywhere (.db files) or MS SQL Server. Open Windows ODBC Data Sources (search "ODBC") and screenshot any DSNs — engine, version, server name, database name.
  • Does it export? On every report screen, look for Export / Save As / Print-to-file. Note which formats: Excel, CSV, PDF, TXT. Find any "Interface," "Export," or "Scheduler" menu.
  • Is there a cloud/sync/API piece? Ask IT or the Comanche rep: any web service, "WProfile" centralized sync, or scheduled auto-export? Get a contact name + email for their Comanche support.
  • Version: Help → About. Screenshot it.

Why this matters: if the DB is readable (ODBC/SQL) or exports can be scheduled, automation can be near-real-time. If it's fully closed, you're limited to manual CSV exports as input — still fine for a v1, but it caps how hands-off the product can be. Know which world you're in before you leave.

B · Export real sample data (your build fuel)

For each item below, export to Excel/CSV and PDF, and save copies to a USB drive and OneDrive. This becomes your test set once you're home — you cannot get it later.

  • A full week of the daily reports the manager actually reads: manager flash / night-audit report, occupancy & RevPAR, room revenue, F&B daily sales, AR aging.
  • Arrivals / departures / in-house guest list.
  • 2–3 sample folios / invoices.
  • Month-end back-office output: P&L, trial balance, GL export.
  • Whatever data feeds the SARA dashboard (see Section D).
  • A guest-profile export (anonymize names if needed — don't carry PII you don't need).

C · Screenshot every screen

Walk each module menu and capture it: Front Office, POS, Inventory Control, Back Office/Accounting, Event Management, Reports. Critically, screenshot the full list of canned report names in the Reports module — that inventory tells you what's already automated vs. what staff still build by hand.

D · Pin down what "SARA" actually is Ambiguous — confirm

Desk research surfaced two different things called SARA, and I can't tell from outside which one your aunt uses. Resolve this on-site:

  • Is SARA part of Comanche (their "Hotel Dashboard & Analytic" layer) or a separate vendor/login (there's an unrelated "S.A.R.A." hotel operations / service-request platform at hotelservices.online)?
  • How do you open it — desktop app, local web page, or internet URL? Screenshot the login + the URL.
  • Where does its data come from? Does it read Comanche's database automatically, or does someone key numbers into it manually? (If manual, that re-keying is itself a wedge.)
  • What does it show, and how fresh is it (live, daily, on-demand)? Who built/maintains it, and what does it cost?

E · Staff pain interviews (Mom Test — past & concrete)

Talk to the GM, front-office manager, and the accountant. Ask about the last time, never the hypothetical. You're hunting for hours-per-week of manual work, not opinions.

  • "Walk me through the last daily report you built by hand — what did you copy, from which screen, into what?"
  • "How long did last month-end close take? What was the annoying part?"
  • "What number does the owner / head office ask for that's a pain to pull?"
  • "What do you re-type out of Comanche into Excel or email every week?"
  • "What does Comanche just not do that you wish it did?"

Get numbers: hours/week, who does it, how often. "3 hours every Monday building the weekly pack" is the kind of evidence that justifies building anything.

F · Commercials & constraints

Quick asks that decide the economics later: annual license/support cost, contract lock-in, who legally owns the data, last time it was upgraded, and whether the aunt would let you use a property as a design partner.

Printable capture checklist

  • ODBC / database engine + version screenshot
  • Comanche version (Help → About) screenshot
  • Export formats available (Excel / CSV / PDF?) noted
  • 1 week of daily reports exported (Excel + PDF) → USB + OneDrive
  • Month-end P&L / GL exported
  • Folio / invoice samples exported
  • Every module menu screenshotted
  • Full canned-report list screenshotted
  • SARA: what it is, URL/app, data source, refresh, owner, cost
  • SARA dashboard fully screenshotted + its source export
  • ≥3 staff interviews done, with hours/week numbers
  • Comanche support contact (name + email)
  • License cost / contract / data-ownership answers

Part 2 — Strategic Brief

What Comanche is

Comanche International is a Thailand-based hospitality-IT company, founded 1996 and listed on Thailand's MAI exchange in 2016 (ticker COMAN). Its PMS is used by ~1,200 hotels and serviced apartments across ~19 countries, heavily concentrated in Thailand and Asia. It's a full-suite, on-premise (local client-server) system — which matches exactly what you described: it lives on their computers, not the cloud. Modules span Front Office (reservations, front desk, housekeeping, billing, night audit), Point of Sale, Back Office accounting (AP/AR/GL), Inventory Control, Event Management, plus a large library of third-party interfaces (channel managers, booking engines, door locks, passport scanners, CRM, yield management).

Strengths

  • Deep, mature, genuinely integrated feature set (30 yrs of edge cases)
  • Built for the Thai/Asian market — local tax, compliance, language, on-the-ground support
  • Large interface library you'd otherwise have to rebuild from scratch
  • Cheap relative to Oracle OPERA; entrenched and sticky
  • Runs locally/offline — resilient to internet outages

Weaknesses

  • Dated client-server architecture; needs on-site servers + someone to maintain them
  • Data effectively locked in a local DB; export is often clunky/manual
  • Reporting is canned operational reports, not flexible BI
  • Little to no native AI; weak remote/mobile access
  • Thin public reputation — 0 verified reviews, no Hotel Tech "Certificate of Excellence"

What SARA likely is

Best read from outside: SARA is the thin analytics/dashboard layer the team glances at — either Comanche's own "Hotel Dashboard & Analytic" product or a separate KPI/operations dashboard. Either way it's a presentation layer, and its usefulness is entirely a function of what data it can see and how fresh that data is. Strength: at-a-glance KPIs without digging through Comanche. Weakness: if it's fed manually or only shows a fixed set of metrics, it's exactly the kind of rigid layer an AI agent replaces. Confirm the data source on-site (Section D) — that one fact reshapes the whole opportunity.

The call: do NOT replace Comanche Blunt

Replacing Comanche means building a PMS. That is the exact saturated trap your own Module 1 research already flagged — generic small-hotel PMS is a graveyard. You'd be fighting a 30-year incumbent with 1,200 live installs, local compliance baked in, and an interface library you'd have to recreate. You would lose, and it wouldn't even prove the thesis. This is the shiny-object pull — name it and kill it.

The thesis test is simple: AI is only a durable edge where it multiplies something only you have. Re-implementing a PMS multiplies nothing — anyone can build CRUD screens. So that's out.

The actual play: an AI layer ON TOP Thesis-positive

Don't change Comanche; read its outputs and automate the managerial work around it — the daily/weekly/monthly reporting, the re-keying, the owner's ad-hoc questions. This is the same "AI-into-existing-workflow" track you're spinning up in the other session, and it's where your real advantages live:

  • Access — your aunt's hotels are a live design partner with real data and a warm intro. Strangers can't get that.
  • Speed — your agent OS lets you turn an export into a polished report pack faster than a vendor ships a feature.

That passes the thesis test: the edge multiplies access + iteration speed, not a commodity anyone can generate.

Decision gates — what makes this real

  1. Data is extractable (Section A comes back "yes, ODBC/SQL or scheduled export").
  2. Confirmed manual-reporting pain — a real hours/week number from staff (Section E).
  3. Aunt will be a design partner and ideally eventually pay (even a token amount = signal).
  4. It generalizes — the same report pack works for other Salil properties, then for the long tail of Thai independents also running Comanche. That's a real wedge market: ~1,200 hotels sharing one data-export pain, ignored by funded incumbents.

Open questions to settle on-site

  • Closed DB or extractable? (the whole thing hinges on this)
  • Is SARA Comanche's layer or a separate tool — and is it fed automatically or by hand?
  • How many hours/week does the team spend on manual reporting today?
  • What does the owner/head office want that's currently painful to produce?
  • Would your aunt let one property be the design partner?
