LifeOS — The Clean Write-Up

Eddie's system, documented 2026-06-12 · Canonical narrative reference · Operational detail lives in the SystemMap (00–06)

1. What LifeOS Is

LifeOS is the operating layer of Eddie's entire life — not an app, but the sum of every system he runs himself through: the Notion dashboards, the Google Calendar, the folders on disk, the daily routines (training, cooking, studying, building), and the data services that observe them (Zepp, Cronometer, Gmail, TradingView).

AgentOS is the engine inside it. Where LifeOS is the body — everything Eddie does — AgentOS is the brain and nervous system: the AI layer that watches the body, tells him what to improve, schedules his time, produces analysis he couldn't produce alone in the hours available, and increasingly acts on his behalf. The two are one system. LifeOS without AgentOS is a well-organized notebook; AgentOS without LifeOS has nothing to think about.

The design philosophy, in one line: Notion is the dashboard, files are the substrate, agents are the engine, the phone is the push channel, and Eddie is the executive.

The Nervous System — coordination + push

  • Coordination files (Misc/AgentOS/): TOKEN_BUDGET.md (normal/light/paused throttle every agent reads first), activity_log.md (audit trail), signals.md (cross-agent events), pending_actions.md (Assistant's inbox), PROTOCOL.md (the rules), time_allocation.md (duration priors).
  • Phone Link bridge — Eddie's largest self-built software: a Windows Python daemon bridging iMessage. Agents push completions/warnings/failures to his lock screen; inbound texts get classified and dispatched. The system's spinal cord — and the substrate the true-agent layer will run on.
  • Model router — Groq → Gemini → Ollama cascade with Claude escalation, built so the mechanical layer survives any pricing regime. The LiteLLM/OpenRouter gateway gives interactive Claude Code a DeepSeek fallback when the Max plan is exhausted.

The Memory — three layers, soon four

  1. Domain markdown — each agent's persistent files (baseline.md, metrics_history.md, WATCHLIST.md, RESEARCH_QUEUE.md, training/injury logs). Append-only where it matters.
  2. Auto-memory — Cowork's persistent memory (user / feedback / project / reference files + MEMORY.md index) carrying cross-session knowledge.
  3. Notion — the human-readable state: tasks, XP ledger, time log, agent run log.
  4. Planned: the recall index — a local embeddings + SQLite FTS index over the entire markdown corpus (every thesis, report, review, note), exposing recall(query) to every agent and to Eddie via Obsidian on top of the same folders. The hippocampus. Free forever, can't be repriced.

Eddie's loop: wake to the briefing → execute against Notion tasks → log workouts/meals/habits → invoke skills on demand (/check-in, /focus, /screen, /workout, /recipe…) → outputs land on disk → Notion and iMessage surface them → Sunday closes the loop with review + next week's schedule.

Every scheduled session follows the same contract: read TOKEN_BUDGET → read standing orders → read coordination files → produce → save to dated folder → log to activity_log → signal if needed → notify Eddie. Skips are silent; failures are flagged.

4. Current State (honest)

What's strong. The architecture itself — file-based coordination, standing orders, token budget culture, append-only archives — is real systems engineering. Output quality is recruiting-grade (audited DCFs, sourced reports, periodized training). Self-monitoring catches real failures. The escape-hatch layer (router + gateway) is designed and mostly built.
What's being pruned (June 2026). School skill tree retired; Gold/loot/achievements economy cut to XP + levels; duplicate Habit Trackers deleted; stale sync-conflict files cleaned.
Known gaps (2026-06-12 audit). The dispatch worker's 72 polls/day is the big token leak (fix: Python poller); TOKEN_BUDGET.md doesn't govern the daily layer; tasks don't verify their own outputs (the Tuesday research job failed silently for two weeks); the notify bridge needs a localhost HTTP endpoint instead of sandbox SQLite writes; health data inputs are the weakest link — the dashboards are better than the data feeding them. The deepest truth: the system produces more than its owner consumes, and the bottleneck is input discipline, not capability.
