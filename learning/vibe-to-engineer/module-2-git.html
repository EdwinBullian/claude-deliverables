Git vs GitHub — they are not the same thing

This is the single most common confusion and clearing it up makes the rest of this module easier.

Git
A program that runs on your computer. It tracks changes to files in a folder — every save, every edit, every history-rewrite — locally. Invented by Linus Torvalds (yes, the Linux guy) in 2005. You can use Git with no internet, no account, no GitHub.
GitHub (and GitLab, Bitbucket, etc.)
A website that hosts copies of Git repositories online so multiple people can sync. It also adds features Git doesn't have natively: Pull Requests, Issues, code review UI, Actions (CI/CD), wikis. Owned by Microsoft. Free for most use.

So when you git push, you're using the Git program on your computer to send your changes to a copy stored on GitHub. The "push" is Git; the "place you're pushing to" is GitHub. Once you internalize this split, everything else makes more sense.

The four places your code lives

This is the model that unlocks Git. Code exists in four locations, and every Git command moves it between them.

  ┌─────────────────┐  edit files   ┌─────────────────┐  git add    ┌─────────────────┐  git commit  ┌─────────────────┐  git push  ┌─────────────────┐
  │  YOU TYPING     │ ────────────▶ │   WORKING       │ ──────────▶ │   STAGING       │ ───────────▶ │   LOCAL REPO    │ ─────────▶ │   REMOTE REPO   │
  │   (editor)      │               │   DIRECTORY     │             │   AREA (index)  │              │   (.git folder) │            │   (GitHub)      │
  └─────────────────┘               └─────────────────┘             └─────────────────┘              └─────────────────┘            └─────────────────┘
                                                                                                                   ◀────── git pull / git fetch ───────
  1. Working directory. The actual files on your disk that you edit in your IDE.
  2. Staging area (a.k.a. "the index"). A holding pen for changes you want in your next commit. You don't commit everything you changed — you commit a curated set.
  3. Local repo. The full history of every commit ever made, stored in a hidden .git folder at the root of your project. This is the actual database.
  4. Remote repo. A copy of the local repo hosted somewhere others can reach — usually GitHub.

Every Git command you'll learn is moving things between those four boxes:

git add file.py        # working dir → staging
git commit -m "..."    # staging → local repo
git push               # local repo → remote
git pull               # remote → local repo (and working dir)
git status             # show what's where
Why staging exists Because you'll often change three files but only want two of them in this commit. Staging is the "ok, these specific changes are what I'm saving" step. If you don't care, git add . stages everything.

Commits and what makes a good one

A commit is a snapshot of your project at a moment in time, with a message describing what changed and why. Every commit has:

  • A unique ID (a long SHA hash like a3f8b1c)
  • An author and timestamp
  • A pointer to the previous commit (forming a chain — the "history")
  • A message you wrote

Good commits are small and focused. One logical change per commit. Bad commits are "fixed stuff" containing 40 file changes across 5 unrelated features.

Writing good commit messages

The widely-cited convention:

Add user avatar to profile dropdown

Avatars are now pulled from /api/users/me and cached for
the session. Falls back to initials if no avatar exists.

Closes #142
  • First line: short summary (~50 chars), imperative mood ("Add" not "Added"), no period.
  • Blank line.
  • Body: why and any context the diff doesn't show.

For solo projects you can be looser. But "fix" as a commit message is a thing you'll regret in six months.

Branches and main

A branch is just a movable pointer to a commit. That's it. Despite all the diagrams and analogies, it's literally a 40-character text file that says "this branch points to commit a3f8b1c."

main (formerly called master) is the default branch in every repo. It's not special — it's just a convention for "the version we trust."

Why branches matter: you don't want to do experimental work directly on main. If it breaks, prod breaks (or other people pulling from main get your broken code). So you branch off, do your work, get it reviewed, and merge back.

# Make a new branch off your current one and switch to it
git checkout -b add-user-avatars
# (modern syntax: git switch -c add-user-avatars)

# See all branches; the * shows where you are
git branch

# Switch back to main
git checkout main
# (or: git switch main)

# Delete a branch you no longer need
git branch -d add-user-avatars

HEAD

HEAD is "where you currently are" — a pointer to the branch you have checked out. When you make a new commit, HEAD (and the branch it's pointing at) move forward to the new commit. That's the whole mystery.

Push, pull, fetch

git push
Send your local commits up to the remote (GitHub). "Publish my work."
git fetch
Download commits from the remote into your local repo, but don't merge them into your working files yet. Read-only update. Useful for "what's new on main without disturbing my current work."
git pull
= git fetch + git merge. Download new commits AND apply them to your current branch. The everyday "sync me up" command.
The classic mistake You make local commits, then run git pull, and Git creates a weird "merge commit" you didn't expect. That's because someone else pushed to the same branch and Git had to merge their work with yours. Fix: use git pull --rebase or set git config --global pull.rebase true once. This keeps history linear instead of creating ugly merge bubbles.

Pull Requests (PRs)

A Pull Request is a GitHub feature, not a Git feature. It's a request to merge a branch into another (usually your branch into main), with a UI for review and discussion before it happens.

Typical flow on a real project:

  1. You branch off main: git checkout -b add-feature
  2. You commit a few times, push the branch: git push -u origin add-feature
  3. On GitHub, you click "Open Pull Request"
  4. Teammates review, leave comments, you push fixes (PRs auto-update)
  5. Once approved + CI passes, you click "Merge" — your branch's commits become part of main
  6. Delete the branch

Solo devs use PRs too, because:

  • It forces you to look at your diff before shipping. You catch dumb stuff.
  • You can run CI (tests, linting) on the PR before merging.
  • It gives you a record of "why was this change made?" linked to issues.
  • You can ask Claude to review the PR.

Merge vs rebase vs squash

Three ways to combine the work of two branches. The difference is what your history looks like afterward.

Merge

Creates a new "merge commit" that has two parents — one from each branch. History looks like a railroad with switches.

  main      A ─── B ─── C ───────── M
                       \           /
  feature               D ─── E ──┘

Pros: preserves the actual history. Cons: can get messy on busy repos.

Rebase

Picks up your branch's commits and replays them on top of the latest main, rewriting history so it looks like you started from there.

  Before:
  main      A ─── B ─── C
                  \
  feature          D ─── E

  After rebase:
  main      A ─── B ─── C
                        \
  feature                D' ─── E'

Pros: clean linear history. Cons: rewrites commit IDs (the D' not D), which is bad if other people have pulled your branch.

Squash

Collapses all your branch's commits into one commit before merging. Common GitHub setting — "Squash and merge" button.

Pros: main history reads as one commit per feature. Cons: you lose the granular history of how the feature evolved.

The honest advice For solo or small-team projects, pick squash merge as your default and stop thinking about it. Use rebase only when you understand why you want it. Don't rebase commits you've already pushed and shared.

Merge conflicts

When two branches change the same lines of the same file, Git can't auto-merge. It marks the file with conflict markers and tells you to fix it by hand:

function greeting() {
<<<<<<< HEAD
  return "Hello!";
=======
  return "Hi there!";
>>>>>>> feature-branch
}

Everything between <<< HEAD and === is the version you currently have. Everything between === and >>> feature-branch is the incoming version. You edit the file to be what you want (delete the markers), then:

git add conflicted-file.js
git commit            # finishes the merge

Modern editors (VS Code, Cursor) show conflict markers as inline buttons: "Accept Current," "Accept Incoming," "Accept Both." Click and move on.

Worktrees

You asked about worktrees specifically. They're great and underused.

Normally a Git repo can only have one branch checked out at a time. If you're working on add-feature and you need to switch to fix-bug, you have to commit (or stash) your work, switch branches, and the working files all change.

A worktree lets you have multiple branches checked out at the same time, in different folders, sharing the same Git history.

# In your main repo folder, on main branch
git worktree add ../my-project-bugfix fix-bug

# Now you have:
#   ./my-project           ← on main, your normal folder
#   ../my-project-bugfix   ← on fix-bug, totally separate working files
# Both share the same .git database, so commits/branches/etc. are visible across them.

Why this is huge for agentic coding: you can let Claude run on the bugfix in one worktree while you keep working on the feature in the other. Superpowers uses worktrees by default precisely for this reason — each parallel agent gets its own isolated workspace so they don't step on each other.

# See your worktrees
git worktree list

# Remove a worktree when done
git worktree remove ../my-project-bugfix

Forks

A fork is your own personal copy of someone else's GitHub repo, under your account. You'd fork when:

  • You want to contribute to an open-source project you don't have write access to. You fork, push to your fork, open a PR from your fork back to theirs.
  • You want to take an existing project as a starting point and make it your own.

Fork vs clone vs branch — easy way to remember:

  • Fork: copy on GitHub (their account → your account).
  • Clone: copy from GitHub to your local machine.
  • Branch: a parallel line of work inside a repo.

.gitignore and what you must never commit

.gitignore is a file at the root of your repo listing patterns Git should pretend don't exist. Examples:

# Secrets — never commit
.env
.env.*
*.key
*.pem

# Dependencies — rebuild from package.json instead
node_modules/
venv/
__pycache__/

# Build output
dist/
build/
.next/

# OS junk
.DS_Store
Thumbs.db

# Editor configs (sometimes)
.vscode/
.idea/
The "I committed an API key" disaster If you push a secret to a public repo, even for 30 seconds, treat it as leaked. Bots scrape new commits within minutes. Rotate the key immediately (regenerate it on the provider's site), then remove it from history with git filter-repo or BFG. Don't just delete the file and commit — the secret still lives in your history.

What to actually do

  1. Set your global config once. 
    git config --global user.name "Eddie Bullian"
git config --global user.email "edwinbullian@gmail.com"
git config --global pull.rebase true
git config --global init.defaultBranch main
  2. For one of your existing projects, do a real PR workflow. Even though you're solo: branch off main, make changes, push the branch, open a PR on GitHub, read your own diff in the GitHub UI, then merge. You'll catch things you wouldn't have on the command line.
  3. Try a worktree. On any repo, run git worktree add ../same-project-experiment main. Open the new folder in a separate window. Realize you now have two parallel Claude sessions you can run simultaneously without them fighting.
  4. Read your reflog once. Run git reflog. This is Git's "undo" record. Almost nothing in Git is truly lost — you can recover from "I just nuked my branch" 99% of the time using this. Knowing it exists is the whole win.
