Reading code > writing code

The biggest unlock between an enthusiastic beginner and an engineer is this: senior engineers read way more code than they write. They read library source when docs are vague. They read past commits to understand why something is the way it is. They read their own diff before pushing.

For you, reading skill is going to be the lever. The model can write code at superhuman speed. The bottleneck is your ability to verify it's correct.

How to read unfamiliar code, fast

  1. Start at the entry point. Find main.py, index.js, app.tsx, or whatever the app boots from.
  2. Read the directory layout first. The folder names tell you the architecture. /api, /components, /lib, /db — already you know a lot.
  3. Read package.json or pyproject.toml. What libraries are in use tells you the tech stack and the assumptions.
  4. Trace one feature end-to-end. Pick one button on the UI, follow it from the click handler all the way to the database. This is the single best exercise.
  5. Don't try to understand everything. Build a fuzzy mental map first, sharp details second.

How to actually review what Claude writes

You said you sometimes don't know what you're doing. This is how to know:

  1. Read the diff before you accept it. Sounds dumb. It's the #1 differentiator. If you don't understand what changed, you can't fix it when it breaks.
  2. Ask "why these specific changes?" Make the model explain each meaningful choice. Why this library? Why this pattern? Why these names?
  3. Watch for over-engineering. AI tools love to add abstraction, configuration, error handling no one needs. If the new code is 3x the size of the problem, ask why.
  4. Run it. Click through the actual feature. Test the edge cases. AI-generated code passes the happy path 95% of the time and the empty-array case 40% of the time.
  5. Read the tests, if any. Tests are the most honest description of what the code is supposed to do.
  6. Run the linter and type checker. Free quality check. Errors mean something.
The "wait, why?" muscle Every time Claude makes a non-obvious choice, ask "why?" Often the answer reveals the choice was actually bad or arbitrary. Sometimes the answer teaches you a real principle. Both are wins.

Refactoring without breaking things

Refactoring = changing the shape of code without changing what it does. Renaming things, extracting functions, restructuring folders, replacing one library with another.

The safety rules:

  1. Tests first. If you don't have tests covering the thing you're about to refactor, write them — at least for the public interface — before changing anything. Otherwise you have no idea if you broke it.
  2. One change at a time. Don't rename, restructure, and rewrite simultaneously. Commit between each step.
  3. Refactor in small, reviewable PRs. A 2000-line "cleanup" PR is unreviewable and high-risk. Three 200-line PRs are tractable.
  4. Don't refactor while adding features. Do one or the other. Mixing them makes both harder to review.

Types — why TypeScript and type hints matter

JavaScript and Python are dynamically typed — variables can hold any type, errors only surface at runtime. TypeScript and Python type hints add an optional layer where you describe what types things should be, and a checker yells before runtime if something's off.

// JavaScript — types are vibes
function add(a, b) {
  return a + b;
}
add(2, "3")  // returns "23". Hope you wanted that.

// TypeScript — checker catches it
function add(a: number, b: number): number {
  return a + b;
}
add(2, "3")  // ❌ Argument of type 'string' is not assignable to parameter of type 'number'
# Python without hints
def get_user(user_id):
    ...

# Python with hints (mypy/pyright check them)
def get_user(user_id: int) -> User | None:
    ...

Why this matters way more for AI-assisted coding: types are your contract with the model. When Claude reads typed code, it knows the shape of every value and generates correct code. When it reads untyped code, it has to guess and guesses wrong. Adding types pays for itself in fewer "why doesn't this work" loops.

Error handling and the "happy path" trap

Beginner code follows the path where everything works. Engineer code asks: what if the network is down? What if the file doesn't exist? What if the API returns an empty list? What if the user types nothing?

The patterns:

# Bad — silently swallow errors. You will regret this.
try:
    do_thing()
except:
    pass

# Bad — catch too broadly. Hides bugs.
try:
    do_thing()
except Exception:
    print("oops")

# Better — catch what you expect, let the rest propagate
try:
    response = requests.get(url, timeout=10)
    response.raise_for_status()
except requests.Timeout:
    logger.warning("API timed out, falling back to cache")
    return cache.get(key)
except requests.HTTPError as e:
    logger.error(f"API error: {e}")
    raise  # let it bubble up to the caller

Two rules:

  • Only catch errors you have a plan for. Otherwise let them crash — at least you'll know.
  • Log enough to debug. "Error" tells you nothing. "Failed to fetch user 42 from /api/users: 503 Service Unavailable" tells you everything.

Logging, observability, "why is it slow"

print() works until it doesn't. Real apps use logging libraries (Python's logging module, JavaScript's pino or winston) that let you:

  • Set log levels (debug, info, warn, error) and filter at runtime.
  • Include structured fields (user_id, request_id) for searching later.
  • Ship logs to a service like Datadog, Honeycomb, or Logflare in production.

Observability — the three pillars

  • Logs — text events. "User 42 logged in." Searchable.
  • Metrics — numbers over time. Requests per second, latency, error rate.
  • Traces — the path a single request took through your system. Which DB query was slow.

For AI apps, add token usage and cost as first-class metrics. You will be very glad you logged this when the bill arrives.

Security basics — the rules you must internalize

The "I leaked an API key" Hall of Shame
  1. Never commit secrets. API keys, passwords, tokens, private keys. Always go in .env, always in .gitignore.
  2. If you commit a secret, rotate immediately. Bots scrape new GitHub commits within minutes. Removing the file doesn't help — it's still in git history.
  3. Use a secrets manager in prod. Vercel/Railway env vars, AWS Secrets Manager, Doppler. Not .env files on the server.

The other rules

  • Never trust user input. Validate everything coming from the client. Assume someone will send "; DROP TABLE users; --
  • Use parameterized queries. Never build SQL by string concatenation. ORMs handle this for you. Raw queries should use ? placeholders.
  • Hash passwords with bcrypt/argon2. Never store plain text. Better: use an auth provider (Clerk, Auth0, Supabase Auth) and don't write auth yourself.
  • HTTPS everywhere. Plain HTTP is dead.
  • Principle of least privilege. Don't run your app as root. Don't give your service account "Owner" permissions. Give the minimum.
  • Keep dependencies updated. npm audit, GitHub Dependabot. Old libraries have known vulnerabilities.

OWASP Top 10

OWASP publishes the top web application security risks. You don't have to memorize it, but skim it once. The categories that bite hobby projects most: injection (SQL/command), broken authentication, exposing sensitive data, insecure direct object references. owasp.org/www-project-top-ten

Performance — when to care

Donald Knuth's quote: "premature optimization is the root of all evil." Translation: don't speed-optimize code before you know it's slow.

The actual workflow:

  1. Make it work.
  2. Make it right (clean, well-tested).
  3. Make it fast — only if needed, only after measuring.

When something is slow, measure first. Don't guess. The bottleneck is almost never where you think.

The hierarchy of slow

If you ever wonder "what's likely slow here," this is the rough order of magnitude:

OperationTime
CPU instruction~1 nanosecond
L1 cache hit~1 ns
RAM access~100 ns
SSD read~100,000 ns (0.1 ms)
Network within data center~500,000 ns (0.5 ms)
DB query (simple)~1–10 ms
API call across the internet~50–500 ms
LLM call~500 ms – 30 seconds

That's why slow apps are almost always slow because of network calls — especially LLM calls. Database queries that hit indexes are fast. The 200ms API call you make in a loop 100 times is what kills you.

Big-O at a hand-wavy level

You don't need formal CS. You need to recognize:

  • Loop inside a loop on a big list = trouble. O(n²). Look for a way to use a dictionary/set lookup instead.
  • Calling an API inside a loop = batch it or parallelize it. Don't make 1,000 sequential network requests if you can make one batched one.
  • Database query inside a loop ("N+1 query problem") = batch with a single query, or eager-load.

What to actually do

  1. Add type hints to one Python project (or convert one JS file to TypeScript). Notice how many bugs the type checker catches without running anything.
  2. Set up structured logging on a project. Switch every print() to a logger with levels. Future you will be grateful.
  3. Set up Dependabot on one of your GitHub repos. Free security wins by just merging the PRs.
  4. Read your own diff before merging a PR. Even when no one else will. Catch one dumb thing per PR — that's a habit that pays for life.
  5. Profile something slow. Pick the slowest thing in one of your apps. Add timing logs around each step. Find where the time actually goes. Almost never where you guess.
