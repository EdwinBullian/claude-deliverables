Terminal vs Command Prompt vs Shell vs Bash

These get jumbled because they're related but they're not the same thing. The cleanest way to think about it:

Terminal (or "terminal emulator")
The window on your screen that shows text and accepts typing. It's a display device. macOS Terminal, iTerm2, Windows Terminal, Warp — all terminals. They don't do anything themselves; they just draw characters.
Shell
The program running inside the terminal that actually interprets what you type. When you type ls and press Enter, the shell is what reads "ls", figures out what command to run, and shows you the output. The terminal is the TV; the shell is the channel.
Bash, Zsh, Fish, PowerShell, cmd.exe
Different shells. Same idea, slightly different syntax and features.
  • bash — the old reliable on Linux. You'll see scripts written in it everywhere.
  • zsh — default on modern macOS. Bash-compatible, nicer features.
  • fish — fancier, friendlier, but not bash-compatible (scripts won't run).
  • powershell — Windows' modern shell. More powerful than cmd, totally different syntax.
  • cmd.exe ("Command Prompt") — the old Windows shell. Avoid unless something forces you.
"Command line" / "CLI"
An umbrella term for "you're typing commands at a shell prompt." Not a specific thing.

On Windows, your best bet is Windows Terminal (the app) running PowerShell or WSL (Windows Subsystem for Linux, which gives you real bash). On Mac, just open Terminal.app or iTerm2 — it runs zsh by default.

Why it matters When a tutorial says "run this in your terminal," they mean the window. When it says "this is a bash script," they're telling you which shell can run it. When it says "command not found," your shell looked for that command and couldn't find it — usually a PATH issue (see below).

Processes, daemons, and background services

A process is just a running program. Open Chrome — that's a process. Run python script.py — that spawns a process. Your OS keeps a list of every process running and gives each one an ID (PID).

Useful commands:

# Mac/Linux — see top resource hogs
top
htop      # nicer version, may need to install

# Linux/Mac — list all processes
ps aux

# Kill a process by PID
kill 12345
kill -9 12345   # force kill (don't ask politely)

# Windows
Get-Process       # PowerShell
tasklist          # cmd
Foreground process
Hooked up to your terminal — output goes to the screen, Ctrl+C kills it. Anything you launch normally.
Background process
Still running, but detached from your terminal. On Unix you put & at the end of a command to launch it in the background: npm run dev &.
Daemon
A process designed to run forever in the background, with no user interface, usually started when your computer boots. Examples: the thing that listens for SSH connections (sshd), the print spooler, the Docker engine. On macOS they're called "launch agents/daemons"; on Windows, "services." The "d" in sshd literally stands for daemon.
Service
Same idea as a daemon, usually the term used on Windows or when talking about software like databases ("the Postgres service is running").

SSH, keys, and remote machines

SSH (Secure Shell) is how you get a shell on a computer that isn't physically in front of you. You type a command, it runs on a server in another building (or another continent), output comes back to your screen.

ssh eddie@my-server.com
# now you're in a shell on that server, as user "eddie"

Password vs key authentication

You can SSH with a password (typed every time, easy to brute force), or with SSH keys (cryptographic, way more secure, also more convenient because you stop typing passwords).

How keys work: you generate a pair — a private key (stays on your laptop, never share) and a public key (you copy this to the server). When you connect, the server challenges your laptop to prove it has the matching private key without sending it over the wire.

# Generate a key pair (do this once per machine)
ssh-keygen -t ed25519 -C "your_email@example.com"

# This creates two files in ~/.ssh:
#   id_ed25519       (private — never share)
#   id_ed25519.pub   (public — paste into GitHub, servers, etc.)

# Copy your public key to a server
ssh-copy-id eddie@my-server.com

This is also how GitHub authenticates you when you push code. When you set up "SSH access" to GitHub, you paste your public key into your GitHub settings.

Never share your private key Anyone with your private key file can log in as you, anywhere it's authorized. Don't commit it to a repo, don't email it, don't paste it in Discord. The .pub file is fine to share; the one without the extension is not.

File paths and the symbols you keep seeing

Absolute path
Starts from the root of the filesystem. On Mac/Linux: /Users/eddie/code/project. On Windows: C:\Users\eddie\code\project. Works from anywhere.
Relative path
Starts from wherever you currently are. code/project means "the code folder inside my current directory."
~ (tilde)
Shortcut for your home folder. ~/Desktop = /Users/eddie/Desktop.
. (single dot)
"The current directory." ./script.sh means "run the script.sh in this folder, not some other one on PATH."
.. (double dot)
"The parent directory." cd .. goes up one level.
/ on Unix, \ on Windows
Path separators. Why they differ is a 40-year-old historical accident. Most modern tools accept both on Windows.

Environment variables and PATH

Environment variables are named values your shell hands to every program it launches. They're how you configure things without hard-coding them.

# See them all
printenv          # Mac/Linux
Get-ChildItem Env: # PowerShell

# Set one for the current shell
export OPENAI_API_KEY="sk-..."     # bash/zsh
$env:OPENAI_API_KEY = "sk-..."     # PowerShell

# Use it in code
import os
key = os.environ["OPENAI_API_KEY"]

The most important environment variable is PATH. It's a list of folders your shell searches whenever you type a command. When you type python, the shell walks through every folder in PATH looking for a file called python and runs the first one it finds.

# See your PATH
echo $PATH
# /usr/local/bin:/usr/bin:/bin:/Users/eddie/.local/bin

# "command not found" usually means: the thing you installed isn't on PATH
The 90% case for "command not found" You installed a tool, but its install folder isn't on PATH. Fix is usually one line added to your shell's config file (~/.zshrc, ~/.bashrc, or PowerShell's profile) like export PATH="$PATH:/new/folder", then restart your terminal.

.env files

For projects, you usually don't export secrets manually. You put them in a file called .env at the project root:

OPENAI_API_KEY=sk-...
DATABASE_URL=postgres://...

Libraries like python-dotenv or Next.js read this file automatically. Critical: add .env to your .gitignore so you never commit secrets. (More on that in Module 2.)

Package managers

Tools that install and manage software. Why one matters: instead of downloading .zip files from sketchy sites, you run one command, the tool fetches it from an official registry, and you can update or uninstall just as cleanly.

brewMac apps & CLI toolsbrew install ffmpeg

npm / pnpm / yarnJavaScript librariesnpm install react

Each language has a package manager that ships with it. You install one tool, that tool installs everything else. When a tutorial says npm install something, they're using the JavaScript package manager.

Ports, localhost, and "what's on 3000"

Every networked program on your computer listens on a numbered port. Ports are like apartment numbers — your IP address gets traffic to the building, the port number gets it to the right apartment.

localhost / 127.0.0.1
"This computer, talking to itself." When you visit http://localhost:3000 in your browser, you're connecting to a program running on your own laptop that's listening on port 3000.
Common ports you'll see
  • 3000 — React, Next.js dev servers
  • 5173 — Vite
  • 8000 / 8080 — Python/Django/FastAPI dev
  • 5432 — PostgreSQL
  • 3306 — MySQL
  • 6379 — Redis
  • 80 / 443 — HTTP / HTTPS in production
"Port already in use"
You tried to start a server on a port, but something else is already there. Either kill the other thing, or use a different port. 
# Find what's on port 3000
lsof -i :3000          # Mac/Linux
netstat -ano | findstr :3000   # Windows

# Kill it (use the PID from above)
kill -9 12345

What to actually do with this

You don't need to memorize this stuff. You need to recognize it so when an error message or tutorial uses the word "daemon" or "PATH" or "process," you're not lost. Three things to actually try:

  1. Generate an SSH key and add it to GitHub. Run ssh-keygen -t ed25519, then paste the contents of ~/.ssh/id_ed25519.pub into GitHub Settings → SSH and GPG keys. Test with ssh -T git@github.com.
  2. Look at your PATH. Run echo $PATH (Mac/Linux) or $env:Path (PowerShell). See what's in there. Notice that the order matters — earlier folders win ties.
  3. Make a .env file. Take one of your existing projects, move a hard-coded API key out into a .env file, add .env to .gitignore, and use os.environ (Python) or process.env (Node) to read it.
