The client–server model

Almost every web app you'll ever build follows this pattern. Internalize this and 80% of architecture conversations become legible.

   ┌─────────────┐                         ┌─────────────┐                ┌─────────────┐
   │             │      HTTP requests       │             │   queries     │             │
   │   CLIENT    │ ──────────────────────▶ │   SERVER    │ ────────────▶ │  DATABASE   │
   │  (browser,  │                          │  (backend   │                │  (Postgres, │
   │   mobile    │ ◀────────────────────── │   API)      │ ◀──────────── │   etc.)     │
   │   app)      │      JSON responses     │             │   results     │             │
   └─────────────┘                         └─────────────┘                └─────────────┘
   "what you see"                       "what does the work"            "where data lives"
  • The client runs on the user's device. It draws the UI and handles interactions.
  • The server runs in the cloud. It does the secret/expensive/authoritative work — talking to the database, checking permissions, calling AI models, etc.
  • The database stores the data persistently. Servers query it; clients usually don't talk to it directly (for security).

Communication goes over HTTP: client sends a request, server sends a response. Usually the response is JSON. That's an API call.

Frontend, Backend, Database — three different jobs

Frontend

Everything the user sees and clicks. Runs in the browser (or as a mobile app). Built with HTML + CSS + JavaScript, almost always inside a framework like React, Vue, Svelte, or Next.js. Lives on the client's device after it's downloaded.

What it's responsible for: layout, animations, form input, fetching data from your API, displaying it. Cannot be trusted with secrets — anyone can open DevTools and read your code.

Backend

The server you control. Runs on a machine somewhere (your laptop in dev, a cloud server in prod). Built with Node.js (Express, Next.js API routes), Python (FastAPI, Django, Flask), Go, Rust, Ruby, Java, etc.

What it's responsible for: authentication, business logic, talking to the database, calling external APIs (Stripe, OpenAI, Anthropic), enforcing permissions. This is where your secrets live — never put API keys in the frontend.

Database

Where data is stored such that it survives a server restart. Two big families covered below.

"Full-stack"

Just means one person (or one framework) handles both frontend and backend. Next.js is the dominant full-stack framework right now — you write frontend pages in the same project as backend API routes.

APIs

An API (Application Programming Interface) is a contract: "here are the operations I expose, here's what to send, here's what you'll get back." In web context, it usually means a set of HTTP endpoints.

Anatomy of an HTTP request

POST https://api.myapp.com/v1/users
Content-Type: application/json
Authorization: Bearer sk-abc123...

{
  "email": "eddie@example.com",
  "name": "Eddie"
}
  • Method (verb): POST, GET, PUT, PATCH, DELETE
  • URL (endpoint): what resource you're hitting
  • Headers: metadata (auth, content type, etc.)
  • Body: the data you're sending (usually JSON)

REST vs GraphQL vs RPC

  • REST — the convention above. URLs are resources, verbs are actions. Everyone knows it. Default choice.
  • GraphQL — one endpoint where the client says exactly what fields it wants. Trade: more flexible queries, more complex setup. Used at scale (Facebook, GitHub).
  • RPC / gRPC — call functions over the network like you would locally. High performance, less common in consumer products.

For 95% of what you'll build, REST + JSON is the right answer until something specific pushes you elsewhere.

Databases — SQL vs NoSQL

SQL (relational) databases

Data in tables with strict schemas, related by foreign keys. Query with SQL (Structured Query Language).

Examples: PostgreSQL (the default these days), MySQL, SQLite (one file, no server — great for small apps).

SELECT users.name, orders.total
FROM users
JOIN orders ON orders.user_id = users.id
WHERE orders.status = 'paid'
ORDER BY orders.total DESC;

NoSQL databases

Various non-relational stores. Different shapes for different jobs:

  • Document stores (MongoDB, Firestore) — store JSON-like blobs.
  • Key-value stores (Redis, DynamoDB) — fast lookups by key. Often used as a cache.
  • Vector databases (Pinecone, Weaviate, pgvector) — store embeddings for AI. Covered in Module 4.
  • Graph databases (Neo4j) — store relationships first-class.
The honest advice Default to Postgres for almost everything. It has JSON support, full-text search, vector extensions (pgvector), and the entire industry's tooling. Don't pick MongoDB because it "sounds modern" — pick it if you have a real reason. SQLite is perfect for small/local apps.

ORMs (Object-Relational Mappers)

Libraries that let you query the database using your programming language's objects instead of writing raw SQL. Examples: Prisma (TypeScript), SQLAlchemy (Python), Drizzle (TypeScript), Django ORM. Useful for productivity. Don't lose sight of the SQL underneath — sometimes raw SQL is the right answer.

Frameworks — what they actually save you

A framework is a pre-built skeleton for an app. It makes a lot of decisions for you (folder structure, routing, conventions) in exchange for letting you skip the boring parts.

The opinionated default for an AI app in 2026 Frontend: Next.js. Backend: either Next.js API routes (if simple) or FastAPI in Python (if you need real ML libraries). Database: Postgres on Supabase or Neon. Deploy: Vercel for the Next side, Railway/Fly for the Python side. This stack will not bottleneck you for a long time.

Languages — when to reach for which

LanguageWhat it's actually for
PythonAI/ML, data science, scripting, backends. The default for anything AI. Slow per-line, but most heavy lifting is in C libraries.
JavaScript / TypeScriptAnything browser-related. Also backends via Node. TypeScript = JavaScript + type checking (use it, always).
HTML / CSSNot "programming" but you need to read both. CSS is harder than people admit.
SQLTalking to databases. You'll touch it even as an AI engineer.
GoBackend services that need to be fast and simple. Docker, Kubernetes, lots of infra is written in Go.
RustPerformance + safety. Tooling (e.g., uv, ruff, biome) is increasingly written in Rust.
C / C++Game engines, OS, embedded, hot performance kernels. You probably won't write it; you'll consume libraries built in it.
Swift / KotlinNative iOS / Android apps respectively.

How to tell what a project is written in: look at the file extensions in the repo. .py → Python. .ts/.tsx → TypeScript. .go → Go. Also look at the lockfile: package.json → Node, requirements.txt or pyproject.toml → Python, Cargo.toml → Rust, go.mod → Go.

Hosting, deployment, and the cloud

"Deploying" = putting your app on a computer with a public address so other people can use it. The spectrum of options, easy to complex:

  1. Platform-as-a-Service (PaaS) — you push code, they run it. Vercel (great for Next.js), Netlify, Railway, Render, Fly.io. Start here. Cost a bit more but save days of work.
  2. Serverless functions — your code runs only when called, scales to zero. AWS Lambda, Vercel Functions, Cloudflare Workers. Great for APIs with spiky traffic.
  3. Containers (Docker) — package your app + all its dependencies into a portable image, run it anywhere. The lingua franca of deployment. Used everywhere from Kubernetes clusters to single-VM hosts.
  4. Raw cloud (IaaS) — you rent a virtual server (AWS EC2, GCP Compute, DigitalOcean droplet) and do everything yourself. Maximum control, maximum work.

The big three clouds

AWS (Amazon), GCP (Google), Azure (Microsoft). They all offer roughly the same services with different names and pricing. AWS is most common, most complex, and most likely to surprise you on the bill. For your projects, you probably don't need to touch any of them — use Vercel/Railway. Learn AWS when an internship forces you to.

Domains, DNS, HTTPS — in plain English

How eddie.dev turns into "my browser loads your site":

  1. You buy eddie.dev from a registrar (Namecheap, Cloudflare, Porkbun).
  2. You configure DNS records pointing the domain at your server's IP address. Common record types:
    • A record — domain → IP address (eddie.dev → 76.76.21.21)
    • CNAME — domain → another domain (www.eddie.dev → eddie.dev)
    • MX — for email
    • TXT — verification, SPF, DKIM (email security)
  3. Someone types eddie.dev. Their computer asks DNS "what's the IP?", gets the answer, sends an HTTP request to that IP.
  4. Your server responds with HTML. Browser renders it.

HTTPS vs HTTP

HTTPS is HTTP encrypted with TLS. Padlock in the address bar. Required for everything modern (browsers will warn or block plain HTTP). Used to be hard to set up; now Let's Encrypt issues free certificates and most hosts (Vercel, Cloudflare) handle it automatically.

What to actually do

  1. Look at one of your existing projects and draw the diagram. Where's the client, where's the server, where's the data, what API calls happen? If you can't draw it, you don't understand it yet.
  2. Open the Network tab in DevTools while using any web app. Watch the actual HTTP requests fly. Look at the URLs, methods, status codes, response payloads. This is the only way to make APIs feel real instead of theoretical.
  3. Build a tiny FastAPI + Next.js app. One endpoint, one page that calls it. Doesn't matter what it does. You'll learn more in two hours of this than a week of reading.
  4. Deploy something to Vercel. Free, takes 5 minutes, demystifies the whole "how does code get on the internet" question.
