What "AI engineering" actually means

There are three jobs that get conflated:

  • ML researcher — invents new models. Reads papers, trains things from scratch. Probably a PhD.
  • ML engineer — trains, fine-tunes, deploys models. Knows PyTorch, data pipelines, GPU clusters.
  • AI engineer — builds products on top of existing foundation models. Writes glue code, designs prompts, sets up RAG, evaluates outputs. This is the one growing fastest and the one most relevant to you.

The AI engineer's core mental shift: the model is a component, not a feature. Your job is to wire it into a system that actually solves the user's problem — give it the right context, the right tools, the right guardrails, and measure whether it's working.

LLM APIs and how the bill works

The big providers expose their models behind HTTP APIs. You send a prompt, you get a response. The major ones:

  • Anthropic — Claude (Opus, Sonnet, Haiku). What's powering this app.
  • OpenAI — GPT-4, GPT-5, o-series reasoning models.
  • Google — Gemini.
  • Open-source via providers — Llama, Mistral, DeepSeek via Together, Groq, Replicate, etc.

A minimal Anthropic call in Python looks like this:

from anthropic import Anthropic

client = Anthropic()  # reads ANTHROPIC_API_KEY from env

msg = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[
        {"role": "user", "content": "What's the capital of France?"}
    ],
)

print(msg.content[0].text)

That's it. Everything else — agents, RAG, tool use — is just sophistication added on top of "send a message, get a response."

Tokens, context windows, and how the bill works

Models don't see characters. They see tokens. A token is roughly 3-4 characters or about ¾ of a word.

  • "Hello" = 1 token
  • "Hello, world!" = 4 tokens
  • A page of text ≈ 500 tokens

Two numbers control your life:

Context window
The maximum tokens the model can see in one call. Claude Sonnet 4.6 = 200K tokens. Long enough for ~150 pages of text. If you exceed it, the API errors or silently drops earlier content.
Pricing — input vs output tokens
You pay separately for input (what you send) and output (what the model generates). Output is usually 3-5x more expensive than input. Example pricing (check current rates): Sonnet might be $3 per million input tokens, $15 per million output tokens.
The cost trap Long conversations + autonomous agents = giant context windows that grow each turn. A poorly-built agent loop can burn through $100 of tokens overnight. Always log token usage; set hard limits per call; use cheaper models (Haiku) for the steps that don't need the big brain.

Prompting as a real discipline

The intuition you've built doing vibe coding is real prompting skill. The formal version just adds structure.

The 5 levers

  1. System prompt — sets the model's persona, role, and standing rules. Sent once.
  2. User message — the actual request.
  3. Examples (few-shot) — show 2-3 input/output pairs of what you want before the real request. Massive quality jump for structured tasks.
  4. Constraints — explicit do's and don'ts. "Output only JSON." "Cite your sources." "If unsure, say so."
  5. Chain of thought — ask the model to think step by step before answering. For tough problems, this just works.

Anthropic's prompt engineering style

Anthropic recommends using XML tags to separate sections of a complex prompt:

<instructions>
Classify the sentiment of the following review as positive, negative, or neutral.
Output only the word.
</instructions>

<example>
Review: "This was the best meal of my life."
Sentiment: positive
</example>

<review>
The pasta was overcooked but the service was friendly.
</review>

Why XML and not Markdown? It's unambiguous — no "is this a heading or a code block" parsing. Models trained with it perform meaningfully better on structured tasks.

Tool use / function calling

By default an LLM can only output text. Tool use (a.k.a. function calling) is how it interacts with the outside world.

You give the model a list of functions it's allowed to call, with their parameter schemas. When the model decides one would help, it returns a structured "I want to call get_weather(city='Paris')" instead of text. Your code executes the function, returns the result back to the model, and it continues.

tools = [{
    "name": "get_weather",
    "description": "Get current weather for a city",
    "input_schema": {
        "type": "object",
        "properties": {
            "city": {"type": "string"}
        },
        "required": ["city"]
    }
}]

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=tools,
    messages=[{"role": "user", "content": "Should I bring an umbrella in Paris?"}]
)

# Model responds with a tool_use block. Your code runs get_weather("Paris"),
# sends the result back, and the model uses it to answer.

This is the foundation of everything agentic. Cowork's connectors, Claude Code's file editing — all tool use under the hood.

RAG — Retrieval-Augmented Generation

Models don't know your private data. They don't know what's in your Notion, your codebase, your company wiki. RAG is the standard pattern for fixing that.

Standard RAG flow:

  1. Ingest: Take your documents (PDFs, web pages, code), split them into chunks (~500 tokens each), turn each chunk into a vector (embedding), store the vectors in a vector database.
  2. Retrieve: When the user asks a question, embed the question into a vector, find the chunks whose vectors are most similar (cosine similarity), return the top 5–10.
  3. Augment: Prepend those chunks to the user's prompt as context.
  4. Generate: Ask the LLM to answer using the provided context.
  User question
       │
       ▼
   ┌──────────┐                ┌──────────────┐
   │ Embed Q  │ ─── vector ──▶ │ Vector DB    │
   └──────────┘                │ similarity   │
                                │ search       │
                                └──────┬───────┘
                                       │ top-K chunks
                                       ▼
                                ┌──────────────┐       ┌──────────┐
                                │  Combine Q + │ ────▶ │   LLM    │ ──▶ Answer
                                │  chunks      │       │          │
                                └──────────────┘       └──────────┘

When NOT to use RAG: if your data fits in the context window, just stuff it in directly. With 200K-token Claude, that's a lot more than people realize.

Embeddings and vector databases

An embedding is a fixed-size list of numbers (a vector) that represents the meaning of a piece of text. The trick: texts with similar meaning have vectors that are close to each other in vector space. This is how "semantic search" works.

embed("dog") ≈ embed("puppy")      # close
embed("dog") ≠ embed("calculator")  # far apart

You generate embeddings via an embedding model (OpenAI's text-embedding-3-small, Voyage AI, Cohere, etc.) — much cheaper than calling an LLM.

Vector databases

Specialized stores designed for "given a query vector, find the K nearest neighbors fast." Without one, comparing one vector against millions takes forever.

  • Pinecone — managed, easy.
  • Weaviate, Qdrant, Chroma — open-source.
  • pgvector — Postgres extension. If you already use Postgres, this is often the right answer (one less service).

Agents and agent loops

An agent is an LLM in a loop with tools and a goal. Pseudocode:

def agent_loop(goal, tools, max_steps=20):
    messages = [{"role": "user", "content": goal}]
    for _ in range(max_steps):
        response = llm.call(messages, tools=tools)
        if response.is_done:
            return response.text
        if response.wants_tool:
            result = run_tool(response.tool_name, response.tool_args)
            messages.append(response)
            messages.append({"role": "tool", "content": result})
    return "Hit step limit"

That's the entire architecture of every agent system, from Cowork to Claude Code to Devin to ChatGPT's web browsing. Everything else is detail — what tools, what prompts, how it handles errors, what guardrails. (Read Anthropic's "Building effective agents" — the canonical reference.)

Multi-agent / subagent patterns

Modern systems often have one "orchestrator" agent that spawns specialized "subagents" for subtasks. Each subagent has its own context window and tool list. Why: cleaner context, parallelizable, lets a small expert do its job without polluting the main thread. Superpowers and Claude Code both lean heavily on this. More in Module 6.

MCP — Model Context Protocol

MCP is an open protocol (designed by Anthropic, now broadly adopted) for connecting AI models to external systems — tools, data sources, prompts. Think of it as "USB-C for AI tools."

Before MCP: every app that wanted to give an LLM access to, say, Slack had to write a custom Slack integration. Every app re-implemented the same thing.

With MCP: someone writes an MCP server for Slack (exposing tools like send_message, list_channels). Any MCP-compatible client (Claude Desktop, Cowork, Cursor, etc.) can connect to that server and instantly gain Slack abilities. The integration is written once.

Architecture:

   ┌─────────────┐       MCP        ┌──────────────┐
   │  Claude     │ ◀──── stdio  ──▶ │ MCP Server   │ ──▶ Slack API
   │  Desktop    │       or         │ (for Slack)  │
   │  (client)   │       HTTP       └──────────────┘
   │             │
   │             │       MCP        ┌──────────────┐
   │             │ ◀────────────── ▶│ MCP Server   │ ──▶ Local files
   │             │                  │ (filesystem) │
   └─────────────┘                  └──────────────┘

Every connector you have in Cowork — Notion, Gmail, Google Calendar, Spotify, Cronometer, TradingView — is an MCP server. The same servers work in Claude Code, Cursor, OpenAI's Codex. Write once, use everywhere.

Why this matters for you You're already using MCP heavily without thinking about it. Once you understand the protocol, building your own MCP server for any private API or tool you care about is a weekend project. Suddenly your AI tools can talk to systems no one's built integrations for. Anthropic has good docs: modelcontextprotocol.io.

Evals — the part nobody talks about

The single thing that separates serious AI engineering from prompt-and-pray: you measure whether your output is actually good.

An eval is a test for AI output. Examples:

  • "For these 50 customer questions, does my support bot's answer match the human-written gold answer at least 80% of the time?"
  • "For these 100 code-review tasks, does Claude catch the planted bug?"
  • "On this set of meeting transcripts, does the summary include all action items?"

The eval can be:

  • Exact match — when there's a single correct answer (classification tasks).
  • Substring / regex — answer must contain certain things.
  • LLM-as-judge — another LLM scores the output against criteria. Surprisingly good, with caveats.
  • Human review — gold standard, slow.

Without evals, you tweak your prompt, it seems better, but you don't actually know if you fixed 3 cases and broke 7. With evals, you know.

Fine-tune vs RAG vs prompt — the decision tree

The flowchart that saves you months:

  1. First: can you fix it with a better prompt? Usually yes. Try few-shot examples, clearer constraints, chain-of-thought. Almost everyone underuses this.
  2. Does the model need access to information it doesn't have? RAG. Private docs, recent data, your codebase.
  3. Does the model need to take actions in the world? Tool use / agents.
  4. Does the model need to behave in a very specific style or format that's hard to specify in a prompt? Maybe fine-tuning. But try lots of few-shot examples first.
  5. Does the model fundamentally not know your domain? Fine-tuning, or maybe a base model swap.

Fine-tuning is what people reach for too early. It's expensive, slow, and the resulting model is frozen — you can't easily update it when your data changes. RAG + good prompting handles 90%+ of real use cases.

What to actually do

  1. Get an Anthropic API key and write a 20-line Python script that makes a real API call. Read the response object structure. This makes the abstraction concrete.
  2. Build a tiny tool-use loop. Give Claude one tool (a calculator, a weather function, anything), put it in a loop, watch it choose to call the tool. Two hours, max.
  3. Build a toy RAG system. Take 5 PDFs, chunk them, embed them with OpenAI embeddings, store in pgvector or Chroma, retrieve + answer. The full loop in one weekend project beats reading about it for a month.
  4. Write your first eval. Pick any AI feature you've built. Make a list of 20 inputs with the answers you'd want. Run your system on all 20. Score the output. Welcome to AI engineering.
  5. Read Anthropic's "Building effective agents." Short, opinionated, by the team that built Claude. anthropic.com/engineering/building-effective-agents
