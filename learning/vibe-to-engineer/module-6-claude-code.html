What Claude Code actually is

Claude Code is a terminal-based agentic coding tool. You launch it in a project directory, type what you want, and it can read files, edit files, run shell commands, search the codebase, run tests — all the things you'd do, in a loop. The agent loop pattern from Module 4, applied specifically to software engineering, with the right tools and prompts wired in.

Three things differentiate Claude Code from "the chat with code blocks" experience:

  1. It runs in your filesystem. Real file edits, real diffs, real test runs.
  2. It can spawn subagents. One main thread orchestrates; specialized subagents handle defined tasks in their own contexts.
  3. It's extensible. Plugins (skills, slash commands, MCP servers, hooks) let you reshape its behavior for your workflow. Superpowers and Deep Trio are the two most influential examples.

Subagents, headless mode, and "agentic coding"

Subagents

A subagent is a sub-instance of Claude that the main agent launches to do a specific job. It gets:

  • Its own fresh context window (no pollution from your chat history)
  • A specific instruction set (e.g., "review this diff for security issues")
  • Optionally, a limited set of tools
  • A scope — it runs, returns one result message, and goes away

Why this is powerful:

  • Context discipline. The main agent doesn't have to dedicate context window to the details of how a particular file was tested — the subagent did that, summarized, and exited.
  • Parallelism. You can dispatch 5 subagents at once on independent subtasks. 5x speedup on the work.
  • Specialization. A "code reviewer" subagent with adversarial instructions gives different (often better) output than asking the main agent to also review its own work.

You can launch a subagent yourself in any Claude Code session by asking the main agent to "spawn a subagent to do X." Modern plugins like Superpowers and Deep Trio do this automatically when their workflows demand it.

Headless mode

Headless mode means Claude Code runs without the interactive UI — non-interactively, scripted, no human in the loop for each step. You give it a task as input, it works to completion, returns the result, exits.

# Headless invocation (one-shot, no chat)
claude -p "Fix all the lint errors in this repo"

This is what lets Claude Code be used as a building block in larger automation — CI/CD pipelines, scheduled jobs, agent armies kicking off other agents.

Quick clarification: "headless" — browsers vs servers

You mentioned "headless servers" and web scraping, so let me untangle the term, because it gets used for two different things:

Headless browser
A real web browser (Chrome, Firefox) running without a visible window. Used for web scraping, automated testing, screenshot generation. The browser still executes JavaScript and renders pages — it just doesn't draw to a screen. Tools: Puppeteer, Playwright, Selenium. This is the one used in scraping.
Headless server / headless mode (generally)
Any program running without an interactive interface — no GUI, no chat, no human stepping through. Driven entirely by inputs/config and outputs to logs or files. Claude Code's headless mode is this kind.

Same word, different domains. Headless = "no visible UI."

Structuring big projects for agentic coding

Agents work better in well-structured projects. The single biggest lever:

The CLAUDE.md file

Drop a CLAUDE.md at the root of your project. Claude Code reads it automatically every session. Put in it:

  • What the project does (1–3 sentences)
  • The tech stack (so it doesn't have to infer)
  • Folder layout and what lives where
  • How to run, test, lint, build
  • Conventions you care about ("we don't use default exports", "all API routes return a standard error envelope")
  • Things to AVOID ("never edit files in vendor/", "don't add new dependencies without asking")

A good CLAUDE.md saves the agent (and you) from re-learning the project every session. Keep it under ~200 lines or it stops being read carefully.

Folder conventions that help agents

  • Clear, descriptive folder names over clever ones. api/, db/, components/ beats core/.
  • Keep related stuff together. Colocate the test next to the file it tests. Colocate styles next to components. Agents (and humans) find things faster.
  • Smaller files over giant ones. A 3,000-line file forces the agent to ingest everything to make one tweak. Twenty 150-line files let it focus.
  • Strong types. Module 5 said this; saying it again. Types are the agent's map.

Use plans and worktrees for anything non-trivial

For features bigger than ~30 minutes of work, don't just say "build feature X." Have the agent plan first (write down the steps, the files to touch, the tests), then execute on the plan with subagents. Use a Git worktree so the work is isolated. This is exactly the philosophy Superpowers and Deep Trio embody.

Superpowers — what it is and how to use it

Built by Jesse Vincent. The repo: github.com/obra/superpowers. The official Claude marketplace install: /plugin install superpowers@claude-plugins-official.

What it actually is

Superpowers is a skill framework: a curated set of skills (small SKILL.md files with instructions) and hooks that automatically trigger at the right moments, plus a methodology the whole system embodies. The methodology is roughly: brainstorm → plan → worktree → subagent-driven implementation with TDD → code review → finish.

The key word is auto-triggering. You don't have to remember to invoke a skill. Once installed, the system watches what you're doing and pulls in the right skill at the right moment.

The skills inside

From the official repo, the skills library covers:

Workflow skills (the spine)

  • brainstorming — Activates before you write code. Refines vague requests through Socratic questions, explores alternatives, presents a design doc in sections for your sign-off.
  • using-git-worktrees — Activates after design approval. Creates an isolated worktree on a new branch, runs setup, verifies clean test baseline.
  • writing-plans — Activates with approved design. Breaks work into 2–5 minute tasks with exact file paths, complete code, verification steps.
  • subagent-driven-development / executing-plans — Activates with plan in hand. Dispatches a fresh subagent per task, with two-stage review (spec compliance, then code quality), or batch execution with human checkpoints.
  • test-driven-development — Enforces RED → GREEN → REFACTOR. Write the failing test, watch it fail, write minimum code, watch it pass, commit.
  • requesting-code-review — Between tasks, runs a code review (often via subagent), reports issues by severity. Critical ones block progress.
  • finishing-a-development-branch — Verifies tests, presents options (merge / PR / keep / discard), cleans up worktree.

Quality / debugging skills

  • systematic-debugging — 4-phase root-cause process. Includes specific techniques like root-cause-tracing, defense-in-depth, condition-based-waiting.
  • verification-before-completion — Don't declare done without proof it actually works.
  • dispatching-parallel-agents — Concurrent subagent workflows.

Philosophy (embedded in every skill)

  • Test-Driven Development always
  • Systematic over ad-hoc
  • Simplicity over cleverness
  • Evidence over claims (verify before saying "done")
How to actually feel Superpowers working Start a session in a Superpowers-equipped repo and ask for something vaguely defined, like "I want to add user avatars." Watch what happens: instead of jumping into code, the brainstorming skill activates and asks clarifying questions. Then the planning skill produces a plan. Then a worktree gets made. Then subagents execute. Each step is the right skill firing at the right time. You can override or skip steps, but the default flow is the value.

The Deep Trilogy — /deep-project, /deep-plan, /deep-implement

Built by Pierce Lamb. Three separate plugins. The Medium write-up: "The Deep Trilogy".

Unlike Superpowers' auto-triggering skills, these are explicit slash commands. You invoke them manually, on the artifacts you've decided you want to work with.

The pipeline

   "I want to build a SaaS platform"
                │
                ▼
        ┌─────────────────┐
        │  /deep-project  │   Decompose vague idea → components
        └─────────────────┘
           │      │      │
           ▼      ▼      ▼
        01-auth  02-bill  03-dash      (each gets its own spec.md)
           │      │      │
           ▼      ▼      ▼
        ┌─────────────────┐
        │   /deep-plan    │   Research → Interview → External LLM Review → TDD Plan → Sections
        └─────────────────┘
           │      │      │
           ▼      ▼      ▼
        ┌─────────────────┐
        │ /deep-implement │   Per section: TDD → Subagent Code Review → Commit
        └─────────────────┘
                │
                ▼
        Working codebase

/deep-project

Input: vague requirements. Output: a folder of focused spec files, each scoped right for planning.

What it does: interviews you about your mental model, identifies logical component boundaries, maps dependencies, generates a spec.md for each component.

Skip this if your idea is already narrowly scoped — go straight to /deep-plan.

/deep-plan

Input: a single spec file. Output: a comprehensive implementation plan with TDD structure, split into self-contained sections, validated by external LLMs.

The phases:

  1. Research — explores your codebase for patterns, searches the web for best practices.
  2. Interview — asks 5–10 clarifying questions to surface hidden requirements.
  3. External LLM review — sends the plan to Gemini and/or ChatGPT for a second opinion (or uses a subagent if no API key).
  4. Integration — Claude reviews external feedback, accepts or rejects suggestions, notes decisions.
  5. TDD structuring — defines test stubs before any implementation.
  6. Section splitting — produces a sections/ folder of self-contained implementation units.

The killer feature: the external LLM review consistently catches blind spots a single model misses. Different context windows surface different issues.

/deep-implement

Input: section files from /deep-plan. Output: working, committed code, one section at a time.

Per section, the cycle is:

  1. TDD implementation — write tests first, then implementation.
  2. Code review — a subagent reviews the staged diff adversarially, looking for edge cases, security issues, logic errors.
  3. Interview — Claude presents findings, you pick which fixes to apply.
  4. Documentation update — section file is updated with "What Was Built."
  5. Atomic commit — one commit per section, clear message, traceable back to the section doc.

Recovery is built in. All three plugins resume from interruption — context limit hit, you stepped away, whatever. Re-run the command on the same artifacts and it picks up where it left off.

Combining Superpowers + Deep Trio (or picking one)

You said you use both at the same time and aren't sure if they conflict. Honest answer: they overlap in places but aren't fundamentally incompatible, and there's a sensible way to think about combining them.

The overlap

ConcernSuperpowersDeep Trio
Brainstorm vague ideabrainstorming skill (auto)/deep-project (explicit)
Plan implementationwriting-plans skill (auto)/deep-plan (explicit, more elaborate)
Execute plansubagent-driven-development/deep-implement
TDD enforcementtest-driven-developmentBuilt into /deep-implement
Code reviewrequesting-code-reviewSubagent in /deep-implement
Worktreesusing-git-worktreesNot opinionated about this

The key philosophical difference

  • Superpowers is invisible and continuous. You don't think about it; skills fire as you work, lifting the whole baseline.
  • Deep Trio is explicit and ceremonial. You decide to invoke /deep-plan; it's a deliberate, multi-phase commitment for that one thing.

Superpowers' planning is solid but lighter than /deep-plan. /deep-plan's external LLM review is the standout feature you don't get elsewhere — Gemini and GPT independently reviewing Claude's plan catches blind spots that "the model reviewing itself" never will.

The recommendation

What I'd do in your shoes Keep both installed. Let Superpowers be your default — it gives you TDD, code review, worktrees, debugging, and the brainstorming skill automatically on everything. When you have something that genuinely deserves a heavy planning investment (a multi-component project, a complex feature, something with non-obvious requirements), explicitly invoke /deep-project or /deep-plan — the external LLM review and the more elaborate planning pipeline are the unique value.

Don't run /deep-implement on a plan that Superpowers also wants to manage — pick one to drive execution per task, or you'll get conflicting skill firings (e.g., both trying to manage commits or run code review). Easiest rule: when you've invoked a /deep- command, follow it through; when you haven't, Superpowers is your default.

Practical workflows by project size

Tiny — a script, a fix, a one-file change (< 30 min)

Just ask. Superpowers' light-touch skills will engage; that's enough. Don't invoke /deep-anything.

Small — one focused feature (1–3 hours)

Let Superpowers' default flow drive: it'll brainstorm → plan → worktree → subagent execution → review → finish. You're along for the ride, approving at checkpoints.

Medium — a feature with multiple parts (a day)

Either let Superpowers drive end-to-end, OR explicitly run /deep-plan @some-spec.md for a deeper plan with external LLM review, then let Superpowers' subagent-driven-development execute. The external review on a medium-complexity feature has saved me from architectural mistakes more than once.

Large — a new multi-component project (multi-day to weeks)

Full Deep Trilogy on day one: /deep-project to decompose, then /deep-plan on each component, then /deep-implement per component. Worktree per component so you can have multiple agents working in parallel. Superpowers skills still fire underneath (TDD, debugging) and that's fine; just don't let Superpowers' planning duplicate /deep-plan's.

Operating tips that actually move the needle

  1. Write a real CLAUDE.md. 30 minutes of work pays for itself within 3 sessions.
  2. Use worktrees as your default for non-trivial work. One per feature/bug. Lets you have multiple agents running without them fighting over the same files. Superpowers does this for you; do it manually otherwise.
  3. Approve plans deliberately. When Superpowers or /deep-plan shows you the plan, actually read it. A 60-second skim catches "you're about to spend 2 hours on the wrong solution."
  4. Use subagents for review. Even outside of plugins, you can ask "spawn a subagent to review this diff adversarially for security and logic bugs." Costs little, catches a lot.
  5. Use cheaper models for routine subagent work. Haiku is fast and cheap; reserve Opus/Sonnet for the planning and complex steps.
  6. Commit small and often. Even with agentic workflows. Atomic commits = recoverable history when something goes sideways.
  7. Don't trust "I'm done." Always verify by running the thing yourself before believing the agent's "all green." This is the verification-before-completion skill made manual.
  8. When stuck, reset context. Long sessions accumulate cruft. Sometimes the right move is "/compact" or starting fresh with a clear restatement of where you are.

What to actually do

  1. Write a CLAUDE.md for your most-used project. Use the structure above. Notice the next few sessions feel sharper.
  2. Run a side-by-side test. Take one feature idea. Build it once with just Superpowers (let auto-skills do their thing). Build a similar one with /deep-plan + /deep-implement driving. Notice the differences in what you got at each phase. You'll quickly form your own opinion about which to reach for when.
  3. Try the worktree workflow manually. git worktree add ../same-project-experiment main. Open both folders in separate windows. Run Claude in both. Realize you can do twice the work without merge pain.
  4. Read Pierce Lamb's "What I Learned Building the Deep Trilogy." Even if you don't write plugins, the insights about why each phase exists will deepen your usage of them. Link.
  5. Read Jesse Vincent's Superpowers release post. Short, opinionated, explains the philosophy. blog.fsck.com/2025/10/09/superpowers
  6. Try writing your own skill or slash command. Both ecosystems make it easy. Once you build one, you understand the whole layer. Anthropic's official skill-creator can help.
